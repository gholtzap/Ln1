# Evaluation Plan

## Goal

Define how to measure whether the product can actually perform useful computer work safely and reliably.

## Why It Matters

Computer control demos can look impressive while failing under normal variation. The product needs repeatable evaluation against real tasks, failures, and safety constraints.

## Desired Capability

The system should be evaluated with representative tasks across apps, websites, files, dialogs, and error conditions. Evaluation should measure completion, safety, recovery, user intervention, and time-to-result.

## Success Criteria

- The product has a clear set of benchmark tasks.
- Success and failure are judged by verified outcomes, not only by action attempts.
- Safety violations are tracked separately from ordinary task failures.
- Evaluations include messy real-world conditions.
- Progress can be measured across development sessions.

## Relationship To The Product

Evaluation keeps the project honest. It helps decide which adapters, actions, and safety features matter most.
